## Context
Asserts are the spec (DEC-0001), so an unachievable or ambiguous assert is a
broken spec. Style rules ("no solutions", "grounding comments") rot unless
checked mechanically — an author (human or agent) cannot be trusted to
self-certify across sessions.

## Decision
Per part, non-negotiable: (1) write /tmp/verify_partN.py with a reference
solution per exercise plus the EXACT asserts copied from the part file;
(2) `pytest /tmp/verify_partN.py -q` must be green; (3)
`python tools/lint_workbook.py part-N.md` must pass; (4) update PROGRESS.md
and delete the /tmp file. If an assert is unachievable, fix the assert —
never compensate by leaking the answer in a hint.

## Alternatives considered
- Author review only — rejected: style drift and broken asserts survive
  self-review, especially post-compaction.
- Checking reference solutions into the repo — rejected: violates the
  zero-solutions rule (RUL-0001); hence the throwaway /tmp file.

## Consequences
Each part costs a verification pass, but shipped asserts are guaranteed
achievable and style is uniform. Bound as RUL-0002.
